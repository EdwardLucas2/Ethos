# Ethos: The Perpetual Contract System (Cycles & Renewals) This document defines how Ethos handles recurring habits, period transitions, and the "parallel state" where a past period is being resolved while a new one is already active. --- ## 1. The Core Architecture: Contract vs. Cycle * **The Contract (The Parent):** The long-term agreement between friends (e.g., "The Gym Rats"). It defines the participants, their individual habits/frequencies, and the shared forfeit. * **The Cycle (The Instance):** A specific time-bound window (Week 1, Week 2, etc.). Each Cycle has its own progress tracking, evidence feed, and resolution outcome. ## 2. Transition Mechanics: The "Overlap" Period When a Cycle ends (e.g., Sunday at 11:59 PM), two things happen simultaneously: 1. **Cycle N enters "Pending Resolution":** Evidence is locked. No new uploads. Majority voting commences to determine the winner/loser of that specific stake. 2. **Cycle N+1 enters "Active":** The progress bars reset. A new "Upload Proof" button appears for the new week. ### The Parallel State UI Strategy * **The Switcher:** On the Contract Detail page, users should be able to toggle between "Current Period" (Active) and "Last Period" (Pending Resolution). * **Unified Dashboard:** The Dashboard should show the status of both. (e.g., "Gym 3x/week: You need 2 more proofs" AND "Last week: 3 reviews needed to settle"). ## 3. Renewal & Exit Logic * **Auto-Renewal by Default:** To maintain momentum, contracts automatically roll into the next Cycle unless a user explicitly chooses to "Opt Out" for the next period. * **The "Opt-Out" Window:** A user can choose to leave a contract at any point, but the exit only takes effect at the *end* of the current active Cycle. * **Adjusting Stakes:** At the start of a new Cycle, the group can vote to change the forfeit (e.g., "Last week was a pint, let's make this week a coffee"). ## 4. User Stories for Renewals **The Habit Builder (The "Don't Break the Chain" User)** * "As a participant, I want my progress bars to reset immediately at the start of the new week so I can upload my Monday morning workout without waiting for my friends to verify last week's Friday workout." * "As a participant, I want to see my 'Streak' (number of successful Cycles in a row) to feel a sense of long-term achievement." **The Settle-Up (The "Accountability" User)** * "As a group member, I want a clear notification that last week's pot is ready to be settled, even while I'm working on this week's goals." * "As a winner, I want to be able to see the 'Pending Resolution' screen for the previous week to ensure everyone pays up, without it cluttering my view of my current goals." **The Quitter (The "Graceful Exit" User)** * "As a participant, I want to be able to signal that I'm leaving the contract at the end of this month so my friends have time to find a replacement or adjust the group." --- ## 5. UI Requirements for the Next Phase 1. **Dashboard Update:** Support for "Multi-Status" contract cards (Active + Pending Resolution). 2. **Contract Overview Update:** Introduce a "Period Switcher" or "History Tab" to access the Pending Resolution view for the previous cycle. 3. **The "Check-In" Screen:** A specialized version of the Resolution screen that acts as a bridge between the old cycle and the new one.
