# Ethos: The "Ready Up" Contract System (Creation & Commitment) This document defines the functional architecture and user requirements for initiating and joining a habit contract. It treats contract formation as a collective "lobby" where the agreement only becomes active once all participants have confirmed their commitments. --- ## 1. Core Logic: The "Lobby" State A contract exists in a non-active **Lobby** state until specific conditions are met: 1. **Creator Initiation:** The person who started the contract triggers the start action. 2. **Individual Sign-off:** Every invited participant has explicitly confirmed their unique commitment. 3. **Temporal Trigger:** The defined Start Date has arrived. --- ## 2. Phase 1: Initiation (The Creator's Flow) ### Features & Actions: * **Contract Definition:** Set the primary parameters: Contract Name, the shared Forfeit (Stake), the Cycle Period (Weekly, Bi-Weekly, Monthly), and the Start Date. * **Participant Management:** Invite friends (up to a limit of 10 total participants). * **Mandatory Initial Commitment:** The creator must define their own habit and frequency as a prerequisite for sending invites. * **Real-Time Lobby Monitoring:** A live view for the creator showing: * Invitation status (Sent vs. Joined). * Habit definitions for each joined participant. * Readiness status for each person: `DRAFTING` (joined but commitment not finalized) or `SIGNED` (ready to start). * **Execution Trigger:** A "Start Contract" action that is gated until at least one other participant has reached the `SIGNED` state. ### User Stories: * "As a creator, I want to see the specific habits my friends are setting so I can ensure the challenge is balanced and fair." * "As a creator, I want to identify which invitees haven't joined yet so I can follow up with them directly." * "As a creator, I want to prevent the contract from starting until everyone has agreed to the terms." --- ## 3. Phase 2: Commitment (The Invitee's Flow) ### Features & Actions: * **Invitation Review:** Access a summary of the contract terms: Creator name, shared forfeit, and start/duration details. * **Peer Intelligence:** View the commitments already made by the creator and other early joiners to inform their own goal setting. * **Goal Specification:** Input their specific habit and frequency for the duration of the contract. * **The Commitment Toggle:** * **Sign Contract:** A definitive action that locks their habit and signals readiness to the group. * **Unsign/Edit:** The ability to revert to a drafting state to modify goals, provided the contract has not yet been started by the creator. * **Decline Path:** A formal way to opt out of the invitation. ### User Stories: * "As an invitee, I want to see what others are doing so my own goal feels appropriately challenging relative to the group." * "As an invitee, I want the flexibility to adjust my goal if I realize I've over-committed before the official start." * "As an invitee, I want to explicitly 'Sign' so there is no ambiguity about my participation." --- ## 4. Functional Components ### A. Real-Time Status List * **Functional States:** * `WAITING`: The invitation is pending. * `DRAFTING`: The user has joined the lobby but has not yet "Signed." * `SIGNED`: The user has locked their commitment and is ready. * **Data Display:** Shows Participant Name and their currently drafted habit/frequency. ### B. Commitment Control * **Logic:** A binary state manager (Sign/Unsign). * **Functional Outcome:** Changes the user's status in the Lobby List and enables/disables the Creator's "Start" trigger. ### C. The Start Trigger * **Logic:** Gated by a "Minimum 2 Signatures" rule (Creator + 1). * **Action:** Transitions the contract from `Lobby` to `Active` and resets all participants' progress for the first cycle. --- ## 5. Dashboard Integration * **Invitation Alert:** A functional notification on the primary dashboard for any pending invitations, serving as the entry point to the Lobby.
